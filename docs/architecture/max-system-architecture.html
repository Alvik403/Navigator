Архитектура системы учета занятий с MAX-ботом, HR-сайтом и сайтом преподавателей

Проектная документация по микросервисной архитектуре, базам данных, потокам данных, портам, Keycloak и технологическому стеку.

Версия 1.0 Дата: 09.06.2026 Backend: Python + FastAPI Auth: Keycloak Нагрузка: до 10 000 пользователей

1. Назначение Документа

Документ описывает целевую архитектуру системы, состоящей из MAX-бота и двух web-сайтов: HR-сайта для учета сотрудников и групп, сайта преподавателей для расписания и занятий, а также MAX-бота для регистрации, уведомлений и быстрых пользовательских сценариев.

Документ не создает приложение и не содержит production-код. Это архитектурный артефакт для последующей реализации.

3. Разбор BPMN-Сценариев

3.1. Регистрация Через MAX-Бота

Пользователь нажимает кнопку регистрации в MAX-боте. Бот передает событие backend-адаптеру, система проверяет пользователя, связывает MAX ID с профилем и Keycloak, сохраняет данные и отправляет информационное сообщение.

BPMN-шагЦелевой компонент
Нажать кнопку регистрацииmax-bot-adapter
Запуск процесса регистрацииuser-service
Проверка данных пользователяuser-service, Keycloak
Выбор направленияgroup-service или справочник направлений
Запись данныхPostgreSQL сервиса-владельца
Вывод сообщенияnotification-service через max-bot-adapter

3.2. Создание Рабочей Группы

HR создает рабочую группу, выбирает сотрудников и преподавателя, редактирует параметры, система проверяет дубли и сохраняет группу.

BPMN-шагЦелевой компонент
Открыть форму создания группыHR web frontend
Получить список сотрудниковuser-service
Выбрать сотрудников и преподавателяHR web frontend, group-service
Проверка дублейgroup-service
Запись группы и составаgroup-service PostgreSQL
Информационное сообщениеFrontend notification + audit event

3.3. Расписание Занятий

Преподаватель или HR выбирает группу, открывает расписание, редактирует занятия. Система проверяет конфликты времени, сохраняет расписание и публикует событие для уведомлений.

3.4. Проведение И Учет Занятия

Преподаватель открывает занятие, отмечает участников и результат. Attendance-сервис сохраняет факт занятия, публикует событие для аудита и уведомлений.

4. Архитектурные Принципы

  1. Backend-сервисы разрабатываются на Python 3.12+ и FastAPI.
  2. Авторизация и SSO централизованы в Keycloak.
  3. Внешний доступ идет только через API Gateway / reverse proxy.
  4. Сервисы stateless и масштабируются горизонтально.
  5. Каждый микросервис владеет своей БД или отдельной PostgreSQL-схемой.
  6. Межсервисные связи строятся через API и события, а не через прямой доступ к чужим таблицам.
  7. Долгие операции, уведомления и синхронизации выполняются асинхронно через брокер сообщений.
  8. Все критичные действия логируются в audit trail.
  9. Система должна иметь мониторинг, трассировку, централизованные логи и алерты.
Практический старт: можно начать с малого числа сервисов или modular monolith, но сразу держать доменные границы так, чтобы позже безболезненно выделить микросервисы.

8. Модель Данных И Базы

Основная транзакционная БД: PostgreSQL. На старте допустим один PostgreSQL-кластер с отдельными схемами на сервис. При росте нагрузки домены можно вынести в отдельные базы.

Правило владения: сервис изменяет только свои таблицы. Другие сервисы получают данные через API или события. Межсервисные связи хранятся как UUID-ссылки, без физических foreign keys между разными БД.

12. Масштабирование На 10 000 Пользователей

10 000 пользователей не означает 10 000 одновременных запросов. Основные пики: утренний вход и просмотр расписания, массовые уведомления, отметка занятий преподавателями, HR-операции и фоновые задачи.

СлойСтартовая конфигурацияМасштабирование
API Gateway2 replicasHPA по RPS/CPU
FastAPI service2-3 replicas на сервисHPA по CPU, latency, queue depth
PostgreSQLPrimary + replicaRead replicas, partitioning, PgBouncer
Redis3 nodesRedis Sentinel или Cluster
RabbitMQ3 nodesQuorum queues
ObservabilityОтдельный контурRetention и sampling

Backend

  • FastAPI через gunicorn + uvicorn-worker или нативный uvicorn.
  • Асинхронные DB-драйверы и HTTP-клиенты.
  • Таймауты на все внешние вызовы.
  • Idempotency keys для повторных команд от MAX и frontend.
  • Pagination и фильтры для списков.
  • Фоновые задачи выносить в отдельные worker pods.

PostgreSQL

  • UUID primary keys.
  • Индексы по внешним идентификаторам, датам и статусам.
  • PgBouncer для контроля соединений.
  • Read replicas для отчетов.
  • Partitioning для audit, notification и attendance history.
  • Backups + point-in-time recovery.

13. Отказоустойчивость И Безопасность

РискМеханизм защиты
MAX API временно недоступенRetry с backoff, DLQ, повторная отправка
Keycloak недоступенКэш JWKS, деградация только для уже выданных JWT
PostgreSQL перегруженPgBouncer, индексы, лимиты запросов, read replicas
Дубли webhook-событийIdempotency table по external event ID
Ошибки уведомленийRetry policy, DLQ, таблица delivery attempts
Массовая рассылка создает пикОчереди, throttling, батчинг
  • TLS для всего внешнего трафика.
  • JWT validation на Gateway и проверка прав в сервисах.
  • RBAC через Keycloak roles + domain-level permissions.
  • PKCE для frontend-клиентов.
  • Service accounts для backend-to-backend.
  • Secrets хранить в Kubernetes Secrets, Vault или облачном Secret Manager.
  • Audit для изменения групп, расписаний, посещаемости и ролей.
  • Rate limiting на login, bot webhook и публичные endpoint.

17. Рекомендации По Реализации Этапами

  • Read replicas.
  • Partitioning больших таблиц.
  • Отдельная аналитическая read-модель.
  • Kafka, если появится потребность в event streaming.

18. Итоговая Рекомендация

Для заданной системы оптимальна архитектура с FastAPI-сервисами, Keycloak как единой системой идентификации, PostgreSQL как основной транзакционной БД, Redis для кэша и rate limiting, RabbitMQ для фоновых событий и уведомлений, а также полноценной наблюдаемостью через OpenTelemetry, Prometheus, Grafana и Loki.

При нагрузке до 10 000 пользователей критично не количество микросервисов, а правильные границы данных, отсутствие синхронных тяжелых цепочек, устойчивые очереди уведомлений, индексы в PostgreSQL, connection pooling и горизонтальное масштабирование stateless-компонентов.
